# 本地 React 预览发布平台 MVP 设计

## 目标

构建一个项目组内部使用的前端预览发布工具。第一版只运行在你的本地电脑上，团队成员可以上传 React/TSX 前端项目，系统自动构建，并生成可分享的预览链接。

MVP 重点验证完整流程：

1. 上传 React 项目压缩包。
2. 在本地隔离环境中构建项目。
3. 保存构建后的静态文件。
4. 生成稳定的预览地址。
5. 可选地通过公网隧道让外部网络访问本地服务。

## 范围

### 包含内容

- 在一台 Windows 本地开发电脑上运行。
- 支持 React 前端项目，重点支持使用 `.tsx` 的 Vite React 项目。
- 通过网页上传 `.zip` 项目压缩包。
- 支持项目列表和版本历史。
- 自动执行 `npm install` 和 `npm run build`。
- 在页面中查看构建日志。
- 为最新成功版本提供静态预览。
- 为每个成功构建版本提供不可变的版本预览链接。
- 提供适合内部团队使用的基础登录能力。
- 使用本地文件系统保存上传文件、构建产物和日志。
- 可选使用 Cloudflare Tunnel、ngrok、frp 或类似工具暴露公网访问地址。

### 不包含内容

- 面向外部用户的 SaaS 多租户平台。
- 计费系统。
- 在线代码编辑。
- AI 代码生成。
- 后端、API、数据库项目部署。
- Git 集成。
- 自定义域名。
- 对象存储或 CDN。
- 生产级复杂用户权限管理。

## 用户角色

- 管理员：管理项目、成员和清理规则。
- 成员：上传项目版本、查看构建日志、复制分享链接。
- 访问者：通过分享链接访问预览页面。

MVP 阶段可以简化管理员和成员的区别，先只做一个登录后的内部用户角色。

## 用户流程

1. 团队成员打开本地管理网站。
2. 成员登录系统。
3. 成员创建项目或选择已有项目。
4. 成员上传包含 React 项目的 `.zip` 压缩包。
5. 后端保存压缩包，并创建构建任务。
6. 构建服务将压缩包解压到临时工作目录。
7. 构建服务执行依赖安装和构建命令。
8. 如果构建成功，系统将 `dist` 或 `build` 目录复制到预览存储目录。
9. 项目的最新预览地址指向新的成功版本。
10. 成员复制最新链接或具体版本链接进行分享。

## 预览链接

本地链接：

```text
http://localhost:3000
http://localhost:3000/p/:projectSlug
http://localhost:3000/p/:projectSlug/v/:versionId
```

公网隧道链接：

```text
https://<tunnel-domain>/p/:projectSlug
https://<tunnel-domain>/p/:projectSlug/v/:versionId
```

项目最新链接始终指向最近一次成功构建的版本。版本链接保持不可变，后续上传新版本不会影响旧版本链接。

## 架构

MVP 可以先实现为一个应用，内部拆成四个模块：

```text
Web 管理界面
  - 项目列表
  - 上传表单
  - 构建状态
  - 构建日志
  - 预览链接复制

API 服务
  - 登录认证
  - 项目管理
  - 版本列表
  - 文件上传
  - 构建任务创建

构建服务
  - 解压 zip
  - 校验项目
  - 在 Docker 中构建
  - 记录日志
  - 发布构建产物

静态预览服务
  - 访问项目最新构建产物
  - 访问指定版本构建产物
  - 支持前端路由回退到 index.html
```

## 推荐技术栈

前端：

```text
React + Vite + TypeScript
```

后端：

```text
Node.js + Express 或 Fastify
```

数据库：

```text
SQLite
```

存储：

```text
本地文件系统
```

构建隔离：

```text
Docker
```

静态预览：

```text
带 SPA fallback 的 Node 静态文件服务
```

为了方便后续迁移到云服务器，存储路径、公开访问地址、构建超时时间等配置都应该通过配置项管理，不要写死在代码里。

## 文件目录结构

运行时数据应该放在源码目录之外，或者至少与应用源码清晰分离：

```text
data/
  app.db
  uploads/
    <project-id>/<version-id>.zip
  builds/
    <project-id>/<version-id>/
      index.html
      assets/
  logs/
    <project-id>/<version-id>.log
  workspaces/
    <version-id>/
```

`workspaces` 是临时工作目录，每次构建完成后可以删除。

## 构建流程

1. 校验上传文件是否为 zip。
2. 将压缩包解压到临时工作目录。
3. 确认项目中存在 `package.json`。
4. 检测包管理器：
   - 如果存在 `pnpm-lock.yaml`，使用 `pnpm`。
   - 如果存在 `yarn.lock`，使用 `yarn`。
   - 否则默认使用 `npm`。
5. 安装依赖。
6. 执行构建命令。
7. 检测构建输出目录：
   - 优先使用 `dist`。
   - 如果没有 `dist`，尝试使用 `build`。
8. 将构建产物复制到 `data/builds/<project-id>/<version-id>/`。
9. 将版本状态标记为成功。
10. 更新项目的最新成功版本。

## 构建限制

MVP 需要保护本地电脑，避免构建任务失控：

- 同一时间只运行一个构建任务。
- 设置构建超时时间，例如 10 分钟。
- 设置上传 zip 大小限制，例如 100 MB。
- 设置解压后项目大小限制，例如 500 MB。
- 构建日志需要限制大小或定期清理。

建议使用 Docker 进行构建隔离。如果第一版原型阶段 Docker 暂时不可用，可以临时使用本机直接构建，但正式 MVP 仍然应该以 Docker 构建为目标。

## 错误处理

常见失败情况需要在页面中清晰展示：

- 缺少 `package.json`。
- 依赖安装失败。
- 缺少 build 脚本。
- 构建命令失败。
- 找不到 `dist` 或 `build` 输出目录。
- 构建超时。
- zip 解压失败。

页面应该展示构建状态和日志，不应该只显示笼统的失败提示。

## 公网访问

MVP 不需要购买服务器。

如果需要让本地网络之外的人访问预览链接，可以使用公网隧道工具将本地服务暴露出去。此时你的本地电脑必须保持开机、联网，并避免进入睡眠状态。

这种方式适合验证产品流程。后续如果需要长期稳定运行，可以将同一套应用迁移到 VPS 云服务器。

## 安全说明

即使是内部团队，上传的项目也可能在依赖安装和构建脚本中执行任意命令，因此系统应该把上传代码视为不可信代码。

MVP 阶段至少需要做到：

- 上传和项目管理需要登录。
- 尽可能在 Docker 中运行构建任务。
- 上传源码、临时工作目录和构建产物必须放在受控目录下。
- 不暴露原始工作目录文件。
- 限制上传大小和构建时长。
- 不在上传项目中保存密钥或敏感配置。

分享预览链接在 MVP 阶段可以先做成知道链接即可访问。如果经常预览敏感页面，可以增加简单访问密码。

## 后续迁移路径

本地 MVP 成功后，可以按以下顺序升级：

1. 将应用迁移到小型 VPS。
2. 使用真实域名和 HTTPS 替代本地公网隧道。
3. 如有需要，将静态构建产物迁移到对象存储。
4. 增加 Git 集成。
5. 增加更细的权限和操作日志。
6. 增加自定义预览域名。

## 成功标准

MVP 达到以下结果即可认为成功：

- 团队成员可以上传 React/Vite 项目 zip。
- 系统可以在没有手动终端操作的情况下完成构建。
- 成员可以查看构建日志。
- 最新预览链接可以打开最近一次成功构建的页面。
- 版本预览链接在后续上传新版本后仍然可访问。
- 在本地电脑在线并开启公网隧道时，其他网络的人可以访问预览页面。
