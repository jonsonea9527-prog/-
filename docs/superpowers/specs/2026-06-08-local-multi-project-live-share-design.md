# 本地多项目实时分享器设计

## 背景

当前已有的 MVP 采用“上传 zip -> 本地构建 -> 生成预览链接”的方式，适合做一次性发布，但不适合当前真实工作流。当前实际使用方式是：

- 在 Cursor 中持续修改 React + Vite 前端项目
- 希望同事通过公网链接实时查看最新页面
- 不希望每次重新上传代码或手动执行多条 PowerShell 命令

因此，下一阶段目标不是继续强化 zip 发布平台，而是做一个运行在本机上的“本地多项目实时分享器”。

## 目标

构建一个本地工具，支持同时管理多个 React + Vite 项目，并为每个项目一键完成：

- 启动本地开发服务器
- 启动公网 tunnel
- 生成并展示公网访问链接
- 在本地代码修改后，依赖 Vite HMR 自动同步到公网预览

## 非目标

本期不做以下内容：

- 不再要求上传 zip 包后构建
- 不做 Cloudflare Tunnel 集成
- 不做固定永久域名
- 不支持 Next.js、CRA 或其他前端框架
- 不做云服务器部署
- 不做用户权限、多人协作、远程项目管理

## 用户场景

### 场景 1：单项目实时分享

用户选择一个本地 React + Vite 项目目录，点击“启动分享”，工具自动启动 `npm run dev` 和 `localtunnel`，随后展示本地地址与公网地址。用户继续在 Cursor 中修改代码，同事通过公网地址实时查看更新。

### 场景 2：多项目并行分享

用户同时维护两个或多个前端项目。工具应允许这些项目同时运行，每个项目拥有自己的端口、本地地址和公网地址，并可独立停止、重启。

### 场景 3：本地工具重启后恢复项目列表

用户关闭工具后重新打开，已添加的项目列表仍然保留，但运行状态默认从“未启动”开始，需要用户重新启动分享。

## 产品形态

第一版采用：

- 本地 Node 服务
- 本地 Web 控制面板

而不是 Electron。原因：

- 实现快
- 容易直接调用本机命令和管理子进程
- 后续如果需要打包成桌面程序，可以在这个基础上再封装

## 架构设计

### 1. Project Store

职责：

- 保存项目元数据
- 支持项目增删改查
- 落盘保存，保证工具重启后仍能读取项目列表

建议保存字段：

- `id`
- `name`
- `projectPath`
- `preferredPort`（可选）
- `createdAt`
- `updatedAt`

不保存运行中进程对象，仅保存静态配置。

### 2. Project Inspector

职责：

- 校验路径是否存在
- 校验目录中是否存在 `package.json`
- 校验是否是 Vite 项目
- 校验是否包含可执行的 `dev` 脚本

识别规则（MVP）：

- `package.json` 存在
- `scripts.dev` 存在
- 依赖或脚本中能识别出 `vite`

失败时返回明确错误，例如：

- 找不到 `package.json`
- 未检测到 `dev` 脚本
- 当前目录不是 React + Vite 项目

### 3. Dev Server Manager

职责：

- 为项目分配端口
- 启动 `npm run dev`
- 监听标准输出
- 解析实际访问地址
- 跟踪进程生命周期

MVP 假设：

- 使用 `npm`
- 在 Windows 环境优先兼容
- 启动命令默认使用项目目录执行 `npm run dev -- --host 0.0.0.0 --port <port>`

说明：

- 强制追加 `--host 0.0.0.0`，保证 tunnel 能访问
- 强制指定端口，避免两个项目抢默认端口

### 4. Tunnel Manager

职责：

- 为每个项目启动 `localtunnel`
- 命令形态为 `npx localtunnel --port <port>`
- 解析输出中的公网地址
- 跟踪 tunnel 进程状态

限制：

- 域名通常不是固定的
- 依赖 `npx` 和网络可用
- 如果 `localtunnel` 启动失败，需要把错误反馈到 UI

### 5. Runtime Supervisor

职责：

- 聚合每个项目的 dev server 和 tunnel 状态
- 对外提供启动、停止、重启接口
- 处理异常退出
- 统一维护运行态数据

每个项目的运行态至少包括：

- `status`: `idle | starting | running | failed | stopping`
- `localUrl`
- `publicUrl`
- `lastError`
- `devPid`
- `tunnelPid`
- `logs`

### 6. Local UI

职责：

- 展示项目列表
- 添加项目
- 启动 / 停止 / 重启单个项目
- 查看本地地址和公网地址
- 复制公网地址
- 查看最近日志和错误状态

UI 形态：

- 卡片式多项目列表
- 每个项目一张卡片
- 顶部提供“添加项目”

每张卡片展示：

- 项目名称
- 本地路径
- 当前状态
- 本地地址
- 公网地址
- 启动 / 停止 / 重启 / 复制链接按钮
- 最近日志摘要

## 数据流

### 添加项目

1. 用户输入或选择本地目录
2. 后端校验项目合法性
3. 保存项目元数据
4. UI 刷新项目列表

### 启动分享

1. Project Inspector 校验项目
2. 分配可用端口
3. Dev Server Manager 启动 Vite
4. 解析本地地址
5. Tunnel Manager 启动 `localtunnel`
6. 解析公网地址
7. Supervisor 更新项目状态为 `running`
8. UI 展示本地地址和公网地址

### 停止分享

1. Supervisor 终止 tunnel 进程
2. Supervisor 终止 dev server 进程
3. 清理运行态
4. UI 状态回到 `idle`

### 重启分享

1. 先执行停止
2. 再执行启动
3. UI 刷新新状态和新公网地址

## 端口策略

为了支持多个项目同时运行，必须自动避让端口。

建议规则：

- 默认从 `5173` 开始找
- 若占用则递增尝试
- 若项目有 `preferredPort` 且未占用，则优先使用

第一版不需要把端口配置开放给用户手工编辑，但后续可以加。

## 错误处理

### 项目校验错误

示例：

- 路径不存在
- 缺少 `package.json`
- 不是 Vite 项目

处理：

- 阻止添加或启动
- 在 UI 中展示明确错误文案

### Dev Server 启动失败

示例：

- `npm install` 未完成
- `npm run dev` 脚本执行失败

处理：

- 项目状态置为 `failed`
- 保留日志输出供用户查看

### Tunnel 启动失败

示例：

- 网络不可用
- `npx` 无法拉取依赖
- `localtunnel` 无法分配地址

处理：

- 项目状态置为 `failed`
- 保留本地 dev server 日志和 tunnel 错误
- 用户可点击“重启”重试

### 运行中进程异常退出

处理：

- 记录退出事件
- UI 提示运行失败
- 不做自动无限重启，避免进入死循环

## 安全边界

第一版仅供内部使用，不做复杂安全体系，但需要明确：

- 工具只能访问用户主动添加的本地目录
- 不执行任意自定义命令，只运行固定的 `npm run dev`
- 公网分享完全暴露当前开发环境页面，用户需自行判断是否可外发

## 测试策略

### 单元测试

- 项目校验逻辑
- 端口分配逻辑
- 项目状态转换逻辑
- 本地配置持久化逻辑

### 集成测试

- 添加合法项目
- 添加非法项目
- 启动单项目
- 同时启动两个项目
- 停止项目
- 重启项目

### 手工验证

- 本地真实 Vite 项目启动
- localtunnel 成功生成公网地址
- 修改代码后公网页面自动更新

## 迭代顺序

1. 本地项目存储与校验
2. 单项目启动 dev server
3. 单项目接入 localtunnel
4. 多项目并行运行
5. Web 控制面板
6. 日志与错误展示

## 后续扩展

如果这版验证通过，下一阶段可以考虑：

- Cloudflare Tunnel 集成
- 固定域名
- 项目分组与备注
- 开机自启动
- 一键打开公网链接
- Electron 封装

## 结论

当前最合适的演进方向不是继续强化 zip 上传发布，而是做一个本地多项目实时分享器。它更贴近当前真实工作流，也能明显减少手工操作成本。MVP 先聚焦 React + Vite、多项目并行、本地一键启动、`localtunnel` 公网分享，先把“比手动开 PowerShell 更方便”这件事做成。
